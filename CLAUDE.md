# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

An unofficial SwiftUI iPhone companion app for Fossil hybrid watches (an independent alternative to the official app — the official app is **not** discontinued as of this writing; it may be retired in the future but that is not certain, so never describe it as retired/discontinued in user-facing copy or App Store metadata). Two hardware families are supported, told apart by the firmware revision string (see `WatchKind.detect`): the **Hybrid HR** (e-ink display, encrypted file protocol, needs the 16-byte auth key) and the older **non-HR Q hybrids** such as the Q Grant (physical hands only, same file protocol but completely unencrypted — no key at all). Several watches can be registered; one is active at a time (`WatchRegistry`), and the UI adapts to the active watch's capabilities. The BLE protocol is an independent implementation of the Fossil `qhybrid` file protocol, cross-checked against Gadgetbridge's `qhybrid` device support (Java, AGPL-3.0) and against real-hardware dumps — the local Gadgetbridge checkout at `/Users/jonathan/Documents/Gadgetbridge` is the cross-reference oracle when implementing or verifying protocol behavior; dumps captured from a real Q Grant are the byte oracles instead. The project is MIT-licensed.

## Commands

The Xcode project is generated from `project.yml` with XcodeGen — edit `project.yml`, never `Hybridge.xcodeproj` directly:

```sh
xcodegen generate    # only needed after changing project.yml
```

Build and test (tests are iOS unit tests, so they need a simulator destination):

```sh
xcodebuild build -project Hybridge.xcodeproj -scheme Hybridge \
  -destination 'platform=iOS Simulator,name=iPhone 17'

xcodebuild test -project Hybridge.xcodeproj -scheme Hybridge \
  -destination 'platform=iOS Simulator,name=iPhone 17'

# single test
xcodebuild test ... -only-testing:HybridgeTests/ProtocolTests/testAlarmTLVFile
```

Prefer `scripts/xbuild.sh build` / `scripts/xbuild.sh test [only-testing-target]` over raw `xcodebuild` in this repo — it filters the compiler-invocation noise down to errors/warnings/results while preserving the exit code (full log in `/tmp/xbuild-<cmd>.log`). Override the simulator with `XBUILD_DESTINATION=…`. The same wrapper runs in CI (`.github/workflows/ci.yml`: `xcodegen generate` → `scripts/xbuild.sh test` on every push/PR).

There is one scheme, `Hybridge`. BLE only works on a real iPhone (the simulator has no Bluetooth), so anything involving the watch itself can only be verified on-device via Xcode; unit tests cover the pure logic only (CRCs, AES vectors, payload/container formats, activity parsing — plus an `ActivityParser` fuzz suite that pins "never traps on malformed input").

## Architecture

The core design is a **serialized request queue over CoreBluetooth**:

- `Sources/BLE/WatchManager.swift` — singleton (`WatchManager.shared`, an `ObservableObject`) owning the CBCentralManager, the connected peripheral, state restoration/auto-reconnect, and the request queue. All CoreBluetooth work happens on `bleQueue`; `@Published` state is updated on main. Only **one request is in flight at a time** — the watch firmware wedges otherwise. Every incoming notification is routed to the current request until it reports `isFinished`. `addLog` logs to the unified system log at `privacy: .private` (payload hex can carry auth-handshake frames, body metrics, contact names); the in-app log screen still shows full text from the in-memory buffer.
- `Sources/Protocol/FossilRequest.swift` — base class for request state machines: a request emits `startData()` to a characteristic, then consumes notifications in `handle(uuid:value:io:)`, talking back through the `RequestIO` protocol (control writes + file-packet streaming). Concrete requests live in `Sources/Protocol/Requests/`.
- `Sources/BLE/WatchActions.swift` — high-level async operations (`initializeWatch`, `authenticate`, `setTime`, `readConfiguration`, watchface install, activity sync…) composed by running requests sequentially via `run(_:)`. Init runs from `WatchManager` (state restoration path), not from the UI. **Every operation that issues a request must hold `WatchSession.exclusive`** (see `Sources/BLE/WatchSession.swift`), and this is now *enforced*, not a convention: `run(_:)` opens with `assert(WatchSession.isHeld)` (read in the caller's task context, before the `bleQueue` hop), so an unheld call trips in debug. Multi-request sequences take it via a thin public wrapper around a `private func …Locked` body; single-request operations take it too (a lone request is exactly what interleaves into someone else's sequence) — including the retry helpers `pushJsonWhenIdle`/`runWhenIdle`, which now acquire the (FIFO) gate rather than polling a fail-fast queue. Acquisition is re-entrant per task, so nested operations are free. Operations shared by both families (`writeConfig`, `setTime`, `setAlarms`, `syncActivity`, hand calibration) branch internally on `WatchRegistry.activeKindSync()`. `installFirmware` guards `activeKindSync() == .hybridHR` before touching the OTA handle — a Q watch must never be flashed with an HR image.
- `Sources/Support/WatchKind.swift` — the watch-family enum + capability flags (`needsAuthKey`, `hasWatchfaces`, `hasHandNotificationConfig`, `hasSubEye`…). UI and actions gate on capabilities, never on the kind itself. Detection run on the 2A26 firmware read; `WatchManager.dispatchInitForKind` then starts the family-specific init (`initializeWatch` vs `initializeQWatch`).
- `Sources/BLE/QWatchActions.swift` — the Q-only operations (plaintext init, config read, vibrate, notification filter, buttons). `Sources/Protocol/QNotificationFilter.swift` and `QButtonConfig.swift` build the Q files in the official iOS app's exact layouts.
- `Sources/Support/WatchRegistry.swift` — the multi-watch roster (`KnownWatch`, active-watch selection, `WatchScoped` per-watch UserDefaults namespacing, `.activeWatchChanged`). **Every per-watch key is a case of `WatchScopedKey`**, and `WatchScoped.perWatchKeys` (what `purge` clears on forget) is derived from `allCases` — so a new scoped setting cannot be added without also being purged. It was a hand-maintained array and did drift: the four `body*` keys were written but never purged, leaving a forgotten watch's age/gender/height/weight behind. Raw values are the on-disk keys — never change one without a migration. `WatchScopedValue<T>` (AppFeatureStores.swift) is the generic Codable-in-UserDefaults store the feature stores are built from; it resolves its key per access, so switching watches needs no reload.
- `Sources/Protocol/PayloadBuilders.swift` / `FossilConstants.swift` — byte-level payload construction, UUIDs, file handles, config item layouts.
- `Sources/Crypto/` — CRC32/CRC32C, AES-CBC/CTR via CommonCrypto, Keychain storage of the 16-byte auth key.
- `Sources/Watchface/` — custom watchface pipeline: photo → 2-bit e-ink encoding (`ImageEncoder`), then `WappBuilder` assembles the `.wapp` container. Text layers are either static (baked into the background image on the phone, `TextLayerRenderer`, any iOS font) or dynamic (`WatchfaceTextLayer.valueSource` — steps/HR/battery/time/date/weekday/calories/active minutes/SpO₂/weather temp/weather condition/chance of rain/UV index, filled in by the watch itself at runtime from `get_common()`): `GlyphAtlas` bakes a per-character glyph atlas in the chosen font/size, and one `widgetText<i>` node per layer (`Resources/fossil_hr_src/widget_text.js`, identical blob copies) draws through `text_layout.json`'s 8 fixed image-node slots. **Hard-capped at 2 dynamic text widgets per face** — a third crashes the face on this firmware (watch reboots on upload); see that directory's README for the two-widget-limit gotcha and the dead ends tried. The four weather sources (`wtemp`/`wicon`/`rain`/`uv`) are the one exception to "always a live `get_common()` field": they only appear there once something asks the phone for them (`req_data`, reverse-engineered from stock complication blobs — see the README's `wtemp` section) and are gone once the shared TTL lapses, so they show "--" until `WeatherProvider` has pushed a reading; **unverified on real hardware**.
- `Sources/Fitness/` — activity file parsing (`ActivityParser`), JSON persistence (`FitnessStore`), Apple Health export (`HealthKitExporter`).
  - `ActivityParser` is **fully bounds-checked and never traps** on malformed input (every read goes through `ByteReader`, whose `peek` guards negative *and* over-range indices; fuzz-tested with ~8k random/truncated inputs). Any structural failure throws `ParseError`; `syncActivity` catches *all* `ParseError` cases (unsupported version or truncated/corrupt) and pauses auto-sync while leaving the file on the watch — it must never trap or wedge into a re-download loop, because the file is the only copy until a successful merge.
  - `HealthKitExporter` records the **actual set of exported timestamps** per watch and coalesces them into `ExportedRange`s only where consecutive timestamps are within the sampling cadence (`contiguousRanges`, `exportCadenceTolerance = 90 s`) — an earlier min/max *span* swallowed intra-span gaps, so a minute missing at export time (a second watch synced late) was marked covered and never written. `merged`/`covers` still guard the inter-run case and the pre-upgrade seed. The `store.save` is **chunked at 5k samples** so the first export after a long history can't OOM the 30 s background task. Every written sample carries `HKMetadataKeySyncIdentifier`, so HealthKit dedups even if the bookkeeping is wrong (restore, reinstall) and the ranges are marked only after the writes land.
  - `FitnessStore` holds health data (heart rate, SpO2, inferred sleep); the watch's copy is deleted after a successful merge, so it is the only copy. **Persistence is a single async path**: `merge` / `setLastSync` / `adoptLegacyData` are `async`, apply the in-memory mutation on the main actor (fast), then call one `persist()` that snapshots (cheap COW) and runs the ~20 MB JSON encode + write on a serial `ioQueue` — never on main. `merge` returns `(count, persisted)`; `syncActivity` gates the on-watch delete on `persisted` (preserving durability via `await`, not a synchronous main-thread write). Dedup is `Set<Int>` of same-watch timestamps (was `Set<String>` of `"ts-uuid"` keys — ~344k allocations/merge). The archive is `.completeFileProtection` (falling back to `…UntilFirstUserAuthentication` when locked) and excluded from backups; `samples` is kept sorted so day lookups binary-search; per-day rollups are cached in `daySummaryCache` and dropped on merge; **an archive that exists but fails to decode is moved to `fitness.corrupt-<date>.json` rather than silently starting empty** — starting empty would let the next persist overwrite months of history. There is exactly one persistence path — do not reintroduce a synchronous `save()`.
- `Sources/Phone/QuietHours.swift` — quiet hours (daily block-everything window). `QuietHours.desiredMode`/`nextBoundary` are pure schedule math (unit-tested); `QuietHoursManager` (BatteryWatcher-shaped singleton, per-watch `WatchScoped` state: `quietSchedule`/`quietModeApplied`/`quietOverride`) resolves schedule + manual override and, only when `connectionState == .ready` and either no init is running or we already hold the session (`WatchSession.isHeld` — the init tails call `evaluate()` while holding `initInProgress` themselves, so a bare flag check silently no-opped them), pushes the block-everything filter variant (`NotificationFilterFile.nightFilter()` HR / `QNotificationFilterFile.nightFilter()` Q) via the day/night-parameterized `setNotificationFilter(night:)`/`setQNotificationFilter(night:)`. `evaluate()` guards `desired != applied` against the persisted `quietModeApplied`, so **anything that writes the day filter to 0x0C00 must invalidate that marker** or the next evaluate no-ops: `setNotificationConfigurations` (HR) and `setQNotificationFilter()` (Q) both call `QuietHoursManager.noteDayFilterApplied()` (sets it to `.day`). Without this, an overnight reconnect ran init — which pushes the day filter — while `applied` still read `.night`, leaving the day filter in place all night; init pushes day, marks it, then calls `evaluate()`, which now re-pushes night. Evaluated from init tails, `periodicMaintenance()`, scene-active, and an intent — never on a timer of its own, since iOS gives background code no precise timers; `BackgroundRefresher.scheduleNext()` sets `earliestBeginDate` to the sooner of the sync interval or the next quiet-hours boundary as a best-effort nudge.
- `Sources/Phone/PhoneFinder.swift` — find-my-phone ring, **foreground only by design**. The app does not declare the `audio` background mode (guideline 2.5.4: background modes only for their intended purpose), so iOS refuses to start a playback session from the background and suspends a playing one on backgrounding. `startOnMain` therefore bails to a local notification when `applicationState == .background`, and a `didEnterBackgroundNotification` observer tears the player down so `isRinging` — which the watch-button toggle branches on — never claims a ring that is inaudible. Don't re-add `audio` to bring background ringing back.
- `Sources/Phone/WorkoutLocationTracker.swift` — GPS distance for watch-initiated workouts. This is the *only* reason the app declares the `location` background mode (`project.yml` `UIBackgroundModes`), and it uses **when-in-use** authorization: `allowsBackgroundLocationUpdates` legitimately *continues* a session **started in the foreground** into the background, so `start()` aborts (with a user notification) when the watch's `workoutApp` event arrives while backgrounded — a new session can't be started from there. `showsBackgroundLocationIndicator = true` (the blue pill) is set for transparency and because App Review looks for it. The real trigger is a watch starting a workout, which a reviewer can't reproduce without hardware, so **`Sources/UI/WorkoutGPSDemoView.swift`** (Settings → Advanced → Workout GPS) drives the same session from a foreground button (`startDemoWorkout`/`stopDemoWorkout`) to make the feature — and the background entitlement — observable on a bare device; point App Review notes at it. The tracker is an `ObservableObject` with main-thread `isRunning`/`liveDistanceMeters` mirrors over its lock-guarded state.
- `Sources/Intents/` — App Intents (sync, vibrate/find watch, set quiet mode, add alarm) plus `FossilShortcuts: AppShortcutsProvider`. All `openAppWhenRun = false` and wait out the same cold-launch reconnect chain as `BackgroundRefresher` via `WatchManager.waitUntilReady`/`waitUntilIdle` (extracted from BackgroundRefresher's original private poll loop — both call sites share it now).
- `Sources/UI/` — SwiftUI screens, thin over `WatchManager`.
- `Widgets/` — the `HybridgeWidgets` home/lock-screen widget extension (steps + watch status, both watch families). It is read-only: `Sources/Support/WidgetSnapshot.swift` (compiled into both targets, Foundation-only) defines a small Codable `WidgetSnapshot` plus display-semantics helpers evaluated against the timeline entry's date, never `Date()`. `Sources/Support/WidgetBridge.swift` (app-only) subscribes via Combine to the already-`@Published` sources (`WatchManager`, `FitnessStore`, `WatchRegistry`) and mirrors a snapshot into `UserDefaults(suiteName: WidgetStore.appGroupID)` on change, debounced and gated by `rendersSameAs` before spending a `WidgetCenter.reloadAllTimelines()` call (budget ~40–70/day). Nothing else about the existing stores changes — this is a one-way mirror, not a migration.

iOS notifications (SMS etc.) reach the watch via ANCS once bonded — the app is not involved in notification delivery.

## Protocol gotchas (verified against the real watch)

- Encrypted file put: **each BLE packet** (including its sequence byte) is AES-CTR encrypted restarting from the same base IV (Java `doFinal` resets the cipher) — not a continuous stream.
- IV construction: `iv[2..7] = phoneRandom[0..5]`, then `iv[7]++`, `iv[9..15] = watchRandom[0..6]`.
- A fresh VerifyPrivateKey handshake is required before every encrypted put.
- Alarm weekday bit order: Sun=0 Mon=1 Tue=2 **Thu=3 Wed=4** Fri=5 Sat=6.
- File packets must never exceed ATT MTU−3 (no long writes); `.wapp` section offsets are absolute, including the 12-byte outer header (code offset = 88).
- Characteristics 3dda0003 and 3dda0005 are write|**indicate** (not notify) — subscribe with `setNotifyValue` anyway; waiting only on `.notify` characteristics hangs the init.
- The watch has **no standard battery service**: battery %/voltage (config item 0x0D) and current step count (0x02) come from an encrypted lookup+get of the configuration file.
- Units config value: 8 = metric base, OR in 4|1 for imperial.
- BLE bonding is triggered *by the watch*, via `[0x02, 0x16]` on 3dda0002 (check status with `[0x01, 0x16]`; reply `[0x03, 0x16, status]`). iOS then shows the system pairing + notifications dialogs. Bonding gives the watch native ANCS (notifications) and AMS (music metadata + volume) — the app is not involved in either after that.
- Config item **0x0001 is the body profile**: 7-byte value `[age years u8][gender u8][height cm u16 LE][weight kg u16 LE][0x00]`, gender `0=non-binary 1=male 2=female`. Fully verified by A→B→A diffing of the 0x0800 config against the official iOS app (height/weight from a 200 cm / 70 kg change; age/gender by toggling one field there-and-back). DOB is stored as **age in whole years**, not a date. The watch firmware uses these for its own daily calorie estimate. Byte 6 is still unknown (always 0). Set via **read-modify-write** (`ConfigItem.bodyProfile(base:…)` patches only the known fields, preserving byte 6) — `WatchManager.setBodyProfile` / the Settings "Body metrics" screen; `reapplyBodyProfileIfNeeded` re-pushes on init when the watch drifts (incl. age advancing on a birthday). Volatile items to ignore when diffing this file: `0x0D` battery, `0x02` steps, `0x04`/`0x08` daily counters, `0x0C` (timestamp+metric), `0x12` (single rolling byte).
- The notification filter (0x0C00) is a hard whitelist for ANCS: no matching entry (or no file at all) = notification silently dropped. The firmware's ANCS key is **CRC32 of the NUL-terminated bundle id** (little-endian). An entry **without** a package-CRC TLV is the catch-all matching every app (the official "all notifications"); an entry with empty icon + `0xC4 = 0` actively blocks that app. Official entry layout (icon *before* priority, priority 0x00 for apps / 0xFF catch-all, vibration 0, enabled-flag 0xC4): `NotificationFilterFile.appendOfficialEntry`, byte-verified against official-iOS-app dumps.

## Q-hybrid gotchas (verified against a real Q Grant, dumps at repo root)

- Family detection: prefix `IV0`/`VA`/`WA` or `charAt(2) == '1'` → HR; else `charAt(6)`: `'0'`/`'1'` → Misfit-era (detected, **unsupported**), `'2'` → the fossil-file protocol we support.
- The Q protocol is the HR file protocol **without any crypto**: plaintext `FilePutRequest`/`FileGetRawRequest`, no `VerifyPrivateKey`, no key prompt. Same transport CRC32 / inner CRC32C.
- **Lookup before get**: a direct get of a file's minor 0 can fail with result code 1 ("invalid operation data").
- Alarms use the legacy 3-byte format (file version ≠ 3, no labels); our `daysMask` is already in the watch's Sun-first bit order.
- Q notifications = hand position + vibration. optional contact-name TLV `0x02` (per-contact call/SMS pairs, matched against the ANCS sender), priority 0x00, movement TLV `0xC2` is **10 bytes** (hour°, min°, subEye −1, duration 10000, trailing −2), `0xC4 = 0`. CRC keying = NUL-terminated bundle id, same as HR (`QNotificationFilterFile`). **Never push an empty filter** — it wipes the config and the watch drops every notification.
- Q button file (0x0600, `QButtonConfigFile`): per button `[0x10/0x20/0x30][entry count][(4-byte header + 00)×count]` (multi-entry = the official "alternate" cycling the sub-eye through alert/time2/alarm/date), then distinct payload blobs (each self-CRC'd, copied verbatim — never synthesize blob bytecode), then a customization section (`[count]` + per payload `[header][0A 00][01 02 01 00]`), trailing CRC32. 
- The "multi-press" payloads report press type in 0x05 event frames: action 2 = single, 3 = double, 4 = long; volume executes phone-side either way (foreground-only, iOS limitation), which is what `QMultiPressStore` maps. Music-control assignment wins the 2/3/4 ambiguity (frames don't identify the source button).
- Hand movement on Q can't step exactly 1° — `MoveHandsRequest(bumpSingleDegree:)` makes it 2°. The sub-eye is physical hand 3 and is zeroed/calibrated along with hour/minute (`hasSubEye`).
- Q init: pairing animation → device info (file versions) → settings triple (goal/vibration/tz) → time → config read. No auth, no pairing check; whether the `[0x02, 0x16]` bonding trigger works on Q firmware is still unverified.
- **Unverified**: quiet hours' Q night filter (`QNotificationFilterFile.nightFilter()`) is a single stub `.app` entry (bundle id `eu.sixpixels.hybridge.quiet`, silent vibration, 0°) rather than an actually-empty file, on the assumption that a real (never-matching) CRC entry blocks all notifications — this has **not** been checked against a real Q Grant yet. Verify by setting a night window, sending a real SMS while it's in effect (expect nothing), then restoring the day filter (expect the alert again) — record the result here once tested.
